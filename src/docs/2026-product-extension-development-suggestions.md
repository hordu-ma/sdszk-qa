# 鲁韵思政大模型 2026 功能拓展开发建议

> 文档定位：功能方向基础稿。合同范围、阶段、模型服务、投标演示和验收以 `2026-luyun-curriculum-pedagogy-development-plan.md` 为主依据。
> 当前仓库仍是问答 MVP；本文功能不代表已经实现。已确认合同覆盖阶段 0–4，阶段 4 必交受控多智能体、多模态和微调评估闭环。

## 结论

鲁韵思政大模型 2026 年升级不应继续停留在单一问答机器人形态，而应升级为面向山东省大中小学思政课教师、教研员和管理者的“思政教学支持工作台”。平台应保留现有轻量问答能力，同时新增知识库增强问答、教学设计生成与诊断、大中小学一体化专题设计、教研工作流、资源库和管理看板。

考虑系统部署在山东师范大学信息中心服务器上，建设路线应坚持“高频轻量实时、复杂任务异步、资源内容缓存、管理端可观测、网页端完整、小程序端轻量”的原则，避免把所有能力都压到实时大模型调用链路上。

## 一、升级背景

当前 MVP 已具备登录认证、主题列表、会话创建、SSE 流式问答、历史会话和审计日志等基础能力，适合作为 2026 升级的底座。

2026 年需要补齐的主要短板包括：

- 现有功能以通用问答为主，尚未形成围绕备课、教研、资源、评价、培训和管理的完整工作流。
- 现有回答主要依赖模型生成，缺少政策文件、教材教参、中心资源和山东地方资源的可追溯知识库支撑。
- 现有功能未区分网页端与小程序端的不同承载边界。
- 现有系统缺少复杂生成任务的异步队列、缓存、额度、降级和管理可观测能力。
- 中心作为省级思政课一体化建设指导机构，需要的不只是教师个人问答工具，还需要区域教研支撑和需求洞察工具。

## 二、产品定位建议

### 2.1 平台定位

产品名称建议沿用“鲁韵思政大模型”，定位升级为：

> 面向山东省大中小学思政课教师、教研员和管理者的省级思政教学智能支持平台。

平台核心价值：

- 为教师提供备课、教学设计、课堂活动、评价工具、教学反思等日常教学支持。
- 为教研员提供集体备课、同课异构、听评课分析、教研报告生成等教研支持。
- 为一体化中心提供资源建设、内容审核、教师需求洞察、区域使用分析和治理能力。
- 为山东省思政课建设沉淀可持续增长的政策、资源、案例和教研知识库。

### 2.2 用户角色

- 一线教师：小学、初中、高中、大学思政课教师。
- 教研员：省、市、县、校各级思政教研人员。
- 中心管理员：负责内容、资源、用户、活动和平台运营。
- 主管部门用户：查看区域数据、资源建设和教师需求趋势。
- 技术管理员：负责模型服务、队列、额度、审计、日志和部署运维。

### 2.3 端侧分工

网页端承载完整工作台：

- 教学设计生成、诊断、编辑和导出。
- 文档上传、资源检索、资源审核。
- 集体备课、教研活动、管理看板。
- 长任务中心和历史成果管理。

小程序端承载高频轻交互：

- 快速问答。
- 简短备课片段生成。
- 收藏、历史、通知、任务提醒。
- 查看异步任务进度和结果摘要。
- 简单材料上传，不承载复杂编辑工作台。

## 三、功能模块开发建议

### 3.1 智能问答中心

功能目标：

- 在现有 SSE 问答基础上，增加场景化、知识库化和可追溯能力。

建议功能：

- 思政教学问答：按小学、初中、高中、大学、一体化、教研等场景切换提示词。
- 政策规范问答：接入教育部、山东省教育厅、思政处、一体化中心相关文件，回答附来源。
- 教材教参问答：按学段、年级、课题检索教材教参内容。
- 山东地方资源问答：接入山东红色资源、齐鲁文化、地方实践案例。
- 多轮教学顾问：围绕一节课持续打磨教学目标、活动、评价和反思。

开发要点：

- 知识库检索优先于大模型生成。
- 回答必须区分“基于资料”和“模型建议”。
- 小程序端限制上下文长度和输出长度。

### 3.2 备课助手

功能目标：

- 将平台从问答工具升级为教师日常备课工具。

建议功能：

- 一键生成教学设计：输入学段、课题、课时、班情、教学目标，生成完整教学设计。
- 议题式教学设计：生成议题、情境、任务链、讨论问题和课堂评价。
- 大中小学一体化教学设计：同一主题生成小学、初中、高中、大学递进式方案。
- 教学目标优化：检查目标是否符合核心素养、是否可观察、可评价。
- 课堂活动设计：生成导入、探究、展示、总结、评价环节。
- 作业与实践任务：生成课后实践、项目化学习、家校协同任务。
- 板书与课件提纲：输出板书结构、PPT 大纲和讲稿框架。

开发要点：

- 复杂教案生成进入异步任务队列。
- 常用模板可预生成或缓存。
- 支持 Word/PDF 导出，但首期可先支持 Markdown/Word 模板导出。

### 3.3 教学设计诊断

功能目标：

- 让教师不仅能生成教案，还能改进已有教案。

建议功能：

- 教案上传诊断：上传或粘贴教学设计，生成结构化诊断结果。
- 目标一致性检查：检查教学目标、活动、评价是否一致。
- 价值引领检查：检查思政导向、表达边界和育人目标。
- 课堂活动有效性检查：判断活动是否有任务、有证据、有反馈。
- 修改建议生成：按问题逐项给出可执行修改方案。
- 评价量规生成：根据教学设计生成形成性评价工具。

开发要点：

- 诊断结果要结构化，便于网页端展示和导出。
- 诊断维度应由中心专家可配置。
- 大文件上传和解析应异步处理。

### 3.4 教研助手

功能目标：

- 支持中心和各级教研组织开展常态化教研。

建议功能：

- 集体备课助手：记录讨论过程，形成共案、分工和修改建议。
- 同课异构方案：围绕同一课题生成不同教学路径和比较维度。
- 听评课记录分析：上传听课记录，生成优点、问题和改进建议。
- 教学反思助手：根据课堂情况生成反思和下一轮优化建议。
- 教研论文/案例初稿：生成教研案例、课题申报、论文结构草稿。
- 教研活动纪要：根据会议记录生成纪要和成果清单。

开发要点：

- 首期以单人使用为主，后续再做多人协同编辑。
- 教研成果应可归档到资源库。
- 需要保留人工修改痕迹和最终版本。

### 3.5 资源库与知识库

功能目标：

- 建设可持续增长的省级思政资源和知识资产。

建议功能：

- 资源检索：统一检索教学案例、课件、微课、政策文件、活动方案。
- 智能标签：自动标注学段、主题、核心素养、适用场景、资源类型。
- 资源推荐：根据教师学段、近期任务和热门主题推荐资源。
- 资源生成：生成课堂材料、任务单、讨论卡、评价表。
- 资源审核流：AI 初审格式、敏感性、适配度，人工终审发布。
- 知识库管理：支持文件上传、解析、切分、向量化、版本更新和停用。

开发要点：

- 知识库必须支持来源引用。
- 资源库和知识库应区分：资源库面向用户下载使用，知识库面向模型检索增强。
- 首期可优先支持 PDF、Word、Markdown、纯文本。

### 3.6 教师研修与培训

功能目标：

- 支持中心组织线上研修、政策学习和教师能力提升。

建议功能：

- 政策学习陪练：围绕政策文件生成学习摘要、问答和测验。
- 案例研修：给出案例材料，教师提交分析后由 AI 反馈。
- 微课程脚本生成：根据培训主题生成讲稿、PPT 大纲和测验题。
- 研修任务管理：发布研修任务，跟踪完成情况。
- 教师研修档案：记录学习、使用、产出和改进轨迹。

开发要点：

- 研修档案涉及个人数据，应有权限边界。
- 测验题生成需要人工审核后发布。

### 3.7 管理端与治理能力

功能目标：

- 为一体化中心提供平台运营、内容治理和区域洞察能力。

建议功能：

- 组织与用户管理：支持省、市、县、校、教师多级组织。
- 内容审核管理：管理资源、知识库、提示词模板和敏感词。
- 使用数据看板：访问量、问答量、任务量、热门主题、活跃学校。
- 需求洞察：聚类教师高频问题，辅助制定教研计划。
- 模型质量抽检：抽查回答、资源生成结果和用户反馈。
- 额度管理：按用户、学校、区域配置调用额度。
- 审计日志：记录登录、问答、生成、下载、审核、管理操作。

开发要点：

- 管理端首期先做只读看板和基础审核，避免过早复杂化。
- 所有模型输出都应能追踪到用户、时间、知识来源和模型配置。

## 四、负载控制与部署建议

### 4.1 交互分层

| 类型 | 建议交互 | 负载策略 |
| --- | --- | --- |
| 高频短问答 | 实时 SSE | 小模型或量化模型，短上下文 |
| 政策/教材问答 | RAG + 实时 | 检索优先，限制引用数量 |
| 教案生成 | 异步任务 | 排队执行，结果缓存 |
| 文档诊断 | 异步任务 | 文件解析后分段处理 |
| 资源标签 | 后台批处理 | 夜间或低峰执行 |
| 管理统计 | 离线计算 | 定时汇总 |
| 小程序请求 | 轻量 API | 更严格限流和输出限制 |

### 4.2 必备技术机制

- 用户级、学校级、IP 级限流。
- 热门问题和模板结果缓存。
- 长任务队列和任务中心。
- 模型调用额度和每日上限。
- 高峰期降级策略。
- 知识库检索结果缓存。
- 模型调用审计和异常监控。
- 管理员可配置提示词模板和模型参数。

### 4.3 推荐部署策略

- API 服务、数据库、对象存储、向量库和模型推理服务分层部署。
- 实时问答和异步生成使用不同队列或不同模型配置。
- 网页端复杂任务走任务中心，小程序端只拉取任务摘要和结果。
- 对接一体化中心平台时优先实现统一身份、统一入口、统一资源链接。
- 正式校内部署、稳定演示和最终验收默认使用 vLLM；Ollama 只作为前期开发、兼容性验证和明确标注的备用 Provider。
- 教学业务通过 ModelGateway 使用逻辑模型名，vLLM/Ollama 的模型 ID、Tokenizer、Chat Template、量化和评测结果分别登记；OpenClaw 不进入鲁韵调用链路。
- `base-spark` 建设 `luyun-int` 集成环境和 `luyun-demo` 稳定演示环境。每个可验收开发节点先部署集成环境并由 `virtus` 验证，通过门禁后以同一镜像晋级稳定演示环境。

## 五、分期建设路线

旧版“2026 上半年/下半年/成熟期”不再作为合同排期，统一采用以下阶段：

| 阶段 | 目标 | 主要范围 |
| --- | --- | --- |
| 阶段 0：联合定义与基线 | 冻结教学与课程论框架、资料、量规、模型和容量基线 | 四层课程教学对象、诊断量规、种子评测集、vLLM/硬件基线、投标故事线 |
| 阶段 1：可信底座与纵向样板 | 建立共用底座并打通一个纵向样板 | RAG 引用、教学成果对象、异步任务、ModelGateway、轻量诊断、版本与导出 |
| 阶段 2：教学设计生产与诊断闭环 | 在样板范围形成可编辑、可诊断、可试点的完整闭环 | 教学设计生产、证据化诊断、逐条修改、专家回归和试点 |
| 阶段 3：一体化特色与生产运营 | 扩展四学段、一体化、资源审核和运营能力 | 四学段与一体化专题、组织权限、资源运营、管理看板；小程序按数据决策 |
| 阶段 4：智能协作与多模态增强 | 完成合同最终增强与 G4 验收 | 受控多智能体、多模态教学证据、微调路径 A/B、综合试点和最终验收 |

合同建议总周期为 41–55 周。每个阶段内的可验收增量均先部署 `luyun-int`，通过 `virtus`、专业质量、安全、迁移和回滚检查后晋级 `luyun-demo`。

## 六、优先级建议

P0 必须优先：

- 知识库问答带引用。
- 教学设计生成。
- 教学设计诊断。
- 异步任务中心。
- 额度、限流、缓存、审计。

P1 建议紧随：

- 大中小学一体化设计。
- 资源库检索与标签。
- 管理端使用看板。
- 小程序轻问答。

P2 后续增强：

- 集体备课。
- 听评课分析。
- 教研论文助手。
- 教师研修档案。

合同阶段 4 硬交付不按 P2 处理：

- 受控多智能体的正式业务流程。
- 图片、音频和短视频的多模态证据定位与人工纠错。
- 微调数据、实验、冻结测试和 Go/No-Go 决策；达到门槛采用路径 A，否则交付路径 B 报告。

## 七、风险与注意事项

- 不建议首期直接做复杂多人协同编辑，容易拉高前端和后端复杂度。
- 不建议所有生成任务都实时返回，容易造成模型服务拥塞。
- 不建议小程序承载完整备课工作台，应保持轻量。
- 不建议无引用地回答政策和教材问题，容易引发准确性风险。
- 不建议把资源库和知识库混为一体，两者面向的用户和技术目标不同。
- 不建议在缺少审核机制前开放自动发布资源。

## 八、下一步落地动作

1. 以详细开发计划冻结阶段 0–4 的功能边界、页面原型和 G0–G4 验收。
2. 明确一体化中心平台网站的登录、用户、组织、资源对接方式。
3. 确认山东师范大学信息中心可提供的 GPU、CPU、内存、存储和网络条件。
4. 明确首批知识库资料范围：政策文件、教材教参、中心资源、山东地方案例。
5. 将功能清单转换为招标参数、验收标准和分阶段交付物，并把 vLLM 默认、Ollama 备用及双环境持续部署写入技术条款。
