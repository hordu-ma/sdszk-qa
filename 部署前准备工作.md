# 部署前准备工作（部署指南）

本文档作为本项目生产部署执行手册，目标是：

- B 服务器承载核心服务（API、PostgreSQL、MinIO、vLLM）。
- A 服务器承载入口层（Nginx + 前端静态），并反向代理到 B。
- 采用“本地打包 → 上传服务器 → 上线切换”的发布方式。

---

## 0. 当前推荐策略（先统一）

基于当前确认信息：B 服务器大模型可被本项目独占使用。

- **默认推荐：Docker 独占部署 vLLM（推荐）**
  - 与项目 `compose` 体系一致，可复现、可回滚、可迁移。
  - 便于统一管理 API + vLLM + 依赖服务。
- **备选：继续使用 systemd 启动 vLLM**
  - 适合已有稳定运维体系。
  - 本项目仅消费已有 `8001` 模型服务。

> 关键约束：同一时刻只能选择一种 vLLM 启动方式（Docker 或 systemd），禁止混用，避免端口 `8001` 和 GPU 资源冲突。

---

## 1. 部署前总检查（门禁）

### 1.1 基础环境

- B 服务器已安装 Docker / Docker Compose。
- 若使用 Docker 跑 vLLM，B 服务器需安装 NVIDIA Container Toolkit，并通过 `docker run --gpus all` 自检。
- A、B 服务器入站/出站白名单已按 `src/docs/服务器白名单.md` 放行。

### 1.2 版本与配置

- 发布包版本已固定（建议记录 Git commit）。
- 生产密钥准备完毕（`JWT_SECRET` 不可用弱口令）。
- `.env` 不入库，线上文件权限建议 `600`。

### 1.3 模型一致性（必须）

- `LLM_MODEL` 必须与 `curl http://<LLM_BASE_URL>/v1/models` 返回 `id` 完全一致。
- 不一致会导致 `404/502` 或对话失败。

---

## 2. 项目中需要调整的内容（上线前）

核心结论：通常不改业务代码，主要改环境变量与部署配置。

### 2.1 代码读取入口（已支持）

- 配置定义：`src/apps/api/config.py` 中 `LLM_BASE_URL`、`LLM_MODEL`。
- 对话调用：`src/apps/api/routes/chat.py` 使用 `settings.LLM_MODEL`。
- 病例生成：`src/apps/api/services/case_generation.py` 使用 `settings.LLM_MODEL`。

### 2.2 必改环境变量（生产）

- `LLM_BASE_URL`：模型服务地址。
- `LLM_MODEL`：模型服务返回的准确模型 `id`。
- `DATABASE_URL`、`MINIO_*`、`JWT_SECRET`、`ENV=production`。

### 2.3 推荐的生产变量示例（B 服务器 API）

```dotenv
ENV=production
DEBUG=false

DATABASE_URL=postgresql+psycopg://<user>:<pass>@postgres:5432/clinic_sim

MINIO_ENDPOINT=minio:9000
MINIO_ACCESS_KEY=<minio_user>
MINIO_SECRET_KEY=<minio_pass>
MINIO_BUCKET=clinic-sim-prod
MINIO_SECURE=false

LLM_BASE_URL=http://vllm:8001
LLM_MODEL=qwen3-30b-a3b

JWT_SECRET=<strong-secret-at-least-32-chars>
```

> 若 API 不与 vLLM 在同一 compose 网络，`LLM_BASE_URL` 请改为 `http://<B_SERVER_IP>:8001`。

## 3. B 服务器部署（默认：Docker 独占 vLLM）

### 3.1 切换前处理（防冲突）

若历史上启用过 systemd vLLM，先停用：

```bash
sudo systemctl disable --now qwen3-30b-a3b.service
sudo systemctl is-active qwen3-30b-a3b.service
```

期望返回：`inactive`。

### 3.2 上传发布包

- 上传内容：后端代码、`src/infra/compose/prod-b.yml`、生产 `.env`、必要脚本。
- 建议目录：`/opt/clinic-sim`（示例）。

### 3.3 启动顺序

```bash
docker compose -f src/infra/compose/prod-b.yml up -d --build postgres minio vllm api
docker compose -f src/infra/compose/prod-b.yml ps
```

### 3.4 数据库迁移（显式执行）

```bash
uv run alembic -c src/apps/api/alembic.ini upgrade head
```

> 迁移执行失败时，不要继续切流量。

### 3.5 B 服务器验收

```bash
curl -sS -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8000/health
curl -sS http://127.0.0.1:8001/v1/models
docker compose -f src/infra/compose/prod-b.yml logs --tail=200 api
docker compose -f src/infra/compose/prod-b.yml logs --tail=200 vllm
nvidia-smi
```

通过标准：

- `health` 返回 `200`。
- `/v1/models` 返回中包含 `qwen3-30b-a3b`。
- API 与 vLLM 日志无持续报错（如 `404/502/timeout/OOM`）。

## 4. B 服务器部署（备选：systemd vLLM + Docker API）

当你选择继续使用已有 systemd 模型服务时，按以下执行：

1. 启动模型服务：

```bash
sudo systemctl start qwen3-30b-a3b.service
sudo systemctl is-active qwen3-30b-a3b.service
curl -sS http://127.0.0.1:8001/v1/models
```

2. 启动数据库与 API（不启动 vllm 容器）：

```bash
docker compose -f src/infra/compose/prod-b.yml up -d --build postgres minio api
```

3. API `.env` 中设置：

- `LLM_BASE_URL=http://<B_SERVER_IP>:8001`（或 API 可达地址）
- `LLM_MODEL=qwen3-30b-a3b`

## 5. A 服务器部署（入口层）

### 5.1 部署内容

- 前端构建产物 `dist`。
- Nginx 增量配置（不覆盖旧主配置）。

### 5.2 生效步骤

```bash
nginx -t
nginx -s reload
```

### 5.3 验收目标

- 前端可访问。
- `/api` 正常转发到 B 服务器。
- 若使用 SSE，对话流式返回正常。

---

## 6. 推荐发布顺序（生产）

1. 本地完成构建、测试与打包。
2. 先部署 B（数据库、对象存储、模型、API）。
3. 完成 B 端健康检查与模型可用性验证。
4. 再部署 A 并切入口流量。
5. 执行端到端冒烟：登录 → 建会话 → 对话 → 提交诊断。

---

## 7. 回滚方案（最小可执行）

### 7.1 代码/镜像回滚

- 回退到上一个发布包（或上一版镜像 tag）。
- 重新 `docker compose up -d`。

### 7.2 配置回滚

- 回退 `.env` 到上一版本。
- 重点回滚：`LLM_BASE_URL`、`LLM_MODEL`、`DATABASE_URL`、`JWT_SECRET`。

### 7.3 模型启动方式回滚（Docker ↔ systemd）

- Docker → systemd：
  - 先停 vllm 容器；再 `sudo systemctl enable --now qwen3-30b-a3b.service`。
- systemd → Docker：
  - 先 `sudo systemctl disable --now qwen3-30b-a3b.service`；再起 vllm 容器。

> 始终保证只保留一种方式占用 `8001`。

---

## 8. 常见故障与快速定位

### 8.1 `curl :8001/v1/models` 连接拒绝

- 说明：模型服务未监听 `8001`。
- 排查：
  - `ss -lntp | grep :8001`
  - `docker compose ... logs vllm` 或 `journalctl -u qwen3-30b-a3b.service -n 200 --no-pager`

### 8.2 API 可启动但对话失败

- 高概率：`LLM_MODEL` 与 `/v1/models` 返回 `id` 不一致。
- 处理：先校准 `LLM_MODEL`，再重启 API。

### 8.3 端口冲突/GPU 冲突

- 高概率：Docker 与 systemd 同时拉起 vLLM。
- 处理：停掉其中一种并重启另一种。
