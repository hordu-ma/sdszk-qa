# 部署前准备工作

## 生产环境大模型变更时需要修改的位置

核心结论：不需要修改业务代码，主要修改环境变量与部署配置。

1. 核心读取入口（代码已支持环境变量）

- 配置定义：`src/apps/api/config.py` 中的 `LLM_BASE_URL`、`LLM_MODEL`
- 对话调用：`src/apps/api/routes/chat.py` 发送 `/v1/chat/completions` 时使用 `settings.LLM_MODEL`
- 随机病例生成：`src/apps/api/services/case_generation.py` 同样使用 `settings.LLM_MODEL`

2. 必改项（无论是否共用模型服务）

- 修改生产环境变量：
  - `LLM_BASE_URL`：指向生产模型服务地址
  - `LLM_MODEL`：填写模型服务 `/v1/models` 返回的准确 `id`

3. 两种部署场景的改法

### 场景 A：共用服务器上现有模型服务（推荐当前场景）

- 只需在本项目生产环境配置中更新：
  - `LLM_BASE_URL`
  - `LLM_MODEL`
- 本项目不需要关心该模型在服务器上的磁盘路径。

### 场景 B：由本项目自行拉起 vLLM

- 在 `src/infra/compose/prod-b.yml` 中设置：
  - `VLLM_MODEL_PATH`（服务器模型路径）
- 同时在 API 环境变量中设置：
  - `LLM_MODEL`（模型服务返回的 model id）

4. 重要一致性要求

- `LLM_MODEL` 必须与 `curl http://<LLM_BASE_URL>/v1/models` 返回的 `id` 完全一致。
- 不一致会导致调用失败（常见为 404/502）。

## Nginx（A 服务器）接入策略：不要覆盖原有配置

结论：不建议直接覆盖正在运行的旧 Nginx 主配置。应采用“增量新增 + 平滑重载”的方式接入本系统。

1. 配置原则

- 不改旧业务 `server` 块，新增独立 `server`（推荐新子域名）或新增独立 `location` 前缀。
- A 服务器仅作为入口层，前端静态资源由 A 提供，`/api` 反向代理到 B 服务器 API。
- 若必须与旧业务同域名共存，优先使用新路径前缀（如 `/clinic-sim/` 或 `/api-clinic/`）避免覆盖 `location /`。

2. 发布方式（零中断）

- 先校验：`nginx -t`
- 再平滑加载：`nginx -s reload`
- 避免使用 `restart`，防止对线上已有系统造成可感知中断。

3. 最小风险上线顺序

- 先用测试域名或灰度流量验证新增配置。
- 验证 API 转发、SSE（如有）、静态资源与证书均正常后，再切主流量。

## A/B 服务器分段部署（两台服务器仅支持直接上传文件）

建议按“B 先、A 后”部署，全部采用“本地打包 → 上传发布目录 → 切换生效”的方式。

### 1. 拆分原则

- B 服务器：核心层，负责 API、PostgreSQL、MinIO、（可选）vLLM。
- A 服务器：入口层，负责 Nginx 与前端静态，并将 `/api` 转发到 B。
- 两台服务器均不依赖在线拉代码，所有发布包都在本地构建后上传。

### 2. B 服务器部署内容（核心层）

- 上传内容：后端代码包、`src/infra/compose/prod-b.yml`、生产 `.env`、必要脚本。
- 必配变量：`DATABASE_URL`、`MINIO_*`、`JWT_SECRET`、`LLM_BASE_URL`、`LLM_MODEL`。
- 若本项目自启 vLLM：额外配置 `VLLM_MODEL_PATH`。
- 启动顺序：`postgres/minio` → `api`（→ 可选 `vllm`）。
- 验证目标：B 能稳定提供 API（如 `http://<B_SERVER_IP>:8000/health` 返回 200）。

### 3. A 服务器部署内容（入口层）

- 上传内容：前端构建产物 `dist`、Nginx 增量配置文件。
- 配置原则：不覆盖旧主配置，新增 `server` 或新增路径前缀并反代到 B 的 API。
- 生效步骤：`nginx -t` 后执行 `nginx -s reload`。
- 验证目标：用户访问 A 可正常加载前端，且 `/api` 请求成功转发到 B。

### 4. 推荐发布顺序

- 第一步：本地完成前后端构建与打包。
- 第二步：先部署 B 并完成后端健康检查。
- 第三步：再部署 A 并切入口流量。
- 第四步：执行端到端冒烟（登录、建会话、对话、提交诊断）。

## 出站白名单与网络连通性（上线前必查）

说明：以下内容与 `src/docs/服务器白名单.md` 保持一致，建议作为上线前门禁检查项。

### 1. B 服务器（核心层）出站放行

- 用途：系统与容器安装（APT + Docker + NVIDIA）
  - 端口：TCP 80/443
  - 域名：`mirrors.tuna.tsinghua.edu.cn`、`nvidia.github.io`、`packages.nvidia.com`
- 用途：Docker 镜像拉取
  - 端口：TCP 443
  - 域名：`docker.mirrors.ustc.edu.cn`
- 用途：模型拉取（ModelScope）
  - 端口：TCP 443
  - 域名：`modelscope.cn`、`www.modelscope.cn`

### 2. A 服务器（入口层）出站放行

- 用途：系统与容器安装（APT + Docker + Nginx）
  - 端口：TCP 80/443
  - 域名：`mirrors.tuna.tsinghua.edu.cn`
- 用途：Docker 镜像拉取
  - 端口：TCP 443
  - 域名：`docker.mirrors.ustc.edu.cn`

### 3. 与部署步骤的对应关系（避免上线中途卡住）

- 若未放行 APT/NVIDIA 相关域名：B 服务器可能无法完成 Docker/NVIDIA 运行时安装。
- 若未放行 Docker 镜像域名：A/B 服务器无法拉取镜像，`docker compose up` 会失败。
- 若未放行 ModelScope 域名：B 服务器无法拉取模型（仅在需要在线拉模型时受影响）。
- 若你采用“本地打包后上传”且服务器不在线拉模型，可将 ModelScope 放行降为“按需”。

### 4. 上线前网络验收清单（建议按顺序执行）

- 第一步（B）：确认可访问 APT、Docker 镜像域名；如需在线拉模型，再确认 ModelScope 域名。
- 第二步（B）：启动 `postgres/minio/api`，校验 `http://<B_SERVER_IP>:8000/health` 返回 200。
- 第三步（A）：确认可访问 APT、Docker 镜像域名，完成 Nginx 配置增量发布。
- 第四步（A）：执行 `nginx -t` 与 `nginx -s reload`，验证前端可访问且 `/api` 转发到 B 正常。
- 第五步（端到端）：执行登录、建会话、对话、提交诊断全流程冒烟。
