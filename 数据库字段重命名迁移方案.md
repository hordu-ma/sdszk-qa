# 数据库字段重命名迁移方案（草案）

## 目标

在不影响现网可用性的前提下，将历史医疗语义字段逐步迁移为思政语义字段。

## 当前现状

- 表名仍为 `cases`，但业务语义已是“主题”。
- 会话仍使用 `case_id` 指向主题。
- 以下字段为历史命名但已语义复用：
  - `chief_complaint` -> 核心诉求
  - `present_illness` -> 场景说明
  - `standard_diagnosis` -> 内部参考答案
  - `case_number` -> 历史序号字段

## 建议迁移策略（分三阶段）

1. 阶段 A（已完成）
- 仅改应用层语义，不改数据库结构。
- 优点：零迁移风险，快速上线。

2. 阶段 B（兼容迁移）
- 在 `cases` 表新增思政语义字段（例如 `core_request`、`scenario_text`、`reference_answer`）。
- 代码双写：新会话写入新旧字段，读优先新字段。
- 编写一次性回填脚本，把历史数据从旧字段拷贝到新字段。

3. 阶段 C（收敛清理）
- 观测稳定后，停止写旧字段。
- 最终迁移中删除旧字段，必要时重命名表（`cases` -> `topics`）。

## 回滚策略

- 阶段 B 任意时点可回滚到“读旧字段”的版本。
- 阶段 C 删除旧字段前必须完成备份；如需回滚，通过数据库备份恢复。

## 风险提示

- 直接重命名表/字段会影响所有 ORM、路由、脚本和迁移链，不建议一步到位。
- 建议先完成 1~2 周双写观察，再执行删除旧字段动作。
