# 生产部署指南

本文档整合以下内容，作为当前唯一的生产部署执行与交接基线：

- A 服务器（前端入口层）部署清单（仅支持上传文件、无 SSH）
- 历史部署手册中的执行步骤、门禁与回滚方案
- 历史部署计划中的方案分支与推进状态

---

## 0. 部署目标与固定拓扑

- A 服务器（入口层）：`172.18.6.117`
  - 已有线上 Nginx 在运行
  - 本项目**不能独占 Nginx**
  - 当前仅支持**直接上传文件**（无 SSH）
- B 服务器（核心层）：`172.18.6.123`
  - 部署 API、vLLM、PostgreSQL、MinIO

架构流向：浏览器 → A(Nginx+静态资源) → `/api` 反代到 B(API:8000) → B 内部访问 vLLM/PostgreSQL/MinIO。

---

## 1. 当前部署策略

### 1.1 推荐路径（优先）

- 方案 A：B 服务器走 Docker 独占 vLLM（网络可拉镜像时）

### 1.2 受限网络备选路径

- 方案 B：B 服务器用 systemd 启 vLLM + Docker 仅跑 API/DB/MinIO
- 方案 C：离线导入镜像后再走方案 A

### 1.3 关键约束

- 同一时刻只能保留一种 vLLM 启动方式（Docker 或 systemd）
- `LLM_MODEL` 必须与 `/v1/models` 返回 `id` 完全一致

---

## 2. 部署前门禁（必须满足）

### 2.1 基础环境

- B 已安装 Docker / Docker Compose
- 若 B 用 Docker 跑 vLLM，已安装 NVIDIA Container Toolkit
- A/B 白名单已按 `src/docs/服务器白名单.md` 放行

### 2.2 配置与安全

- 生产密钥已准备完毕（`JWT_SECRET` 为强密钥）
- 线上 `.env` 不入库，权限 `600`
- 发布版本已固定（建议记录 commit）

### 2.3 模型一致性

- `curl http://<LLM_BASE_URL>/v1/models` 中存在目标 `id`
- `.env` 中 `LLM_MODEL` 与该 `id` 完全一致

## 3. A 服务器部署（仅上传文件，无 SSH）

> 适用前提：A 服务器无法 SSH，只能上传文件到指定目录；Nginx 由运维侧统一管理。

### 3.1 本地构建前端

在本地执行：

```bash
cd src/apps/web
npm ci
npm run build
```

构建产物目录：`src/apps/web/dist`。

### 3.2 上传静态资源（仅上传）

- 将 `dist` 目录内文件上传到 A 服务器静态目录（示例：`/data/www/clinic-sim/dist`）
- 注意上传时不要多包一层 `dist/dist`
- 建议保留上一版本静态文件备份，便于快速回滚

### 3.3 Nginx 增量并入（由运维执行）

- 使用模板：`src/infra/compose/nginx/nginx.conf`
- 此文件是**增量 location 片段**，必须并入已有 `server {}`，不要替换主配置
- 关键路由要求：
  - `/` 指向 A 上静态目录
  - `/api/` 反代到 `http://172.18.6.123:8000`
  - `/api/chat` 与 `/api/chat/` 保持 SSE 禁缓冲

### 3.4 无 SSH 场景下的生效动作

你本人无法执行命令，需运维在 A 服务器完成：

```bash
nginx -t
nginx -s reload
```

### 3.5 A 服务器验收（你可在本地浏览器完成）

- 访问业务域名，页面可正常打开
- 登录后接口请求正常（无 404/502/CORS）
- 对话接口 `/api/chat` 可流式返回
- 前端资源加载状态为 200/304

### 3.6 A 服务器回滚（仅上传也可完成）

- 直接回传上一版静态文件
- 运维回滚 Nginx 增量片段并 reload

## 4. B 服务器部署（核心层）

## 4.1 方案 A：Docker 独占 vLLM（推荐）

### 4.1.1 切换前处理

```bash
sudo systemctl disable --now qwen3-30b-a3b.service
sudo systemctl is-active qwen3-30b-a3b.service
```

### 4.1.2 启动服务

```bash
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml up -d --build postgres minio vllm api
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml ps
```

### 4.1.3 数据库迁移

```bash
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml exec api uv run alembic -c src/apps/api/alembic.ini upgrade head
```

### 4.1.4 验收

```bash
curl -sS -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8000/health
curl -sS http://127.0.0.1:8001/v1/models
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml logs --tail=200 api
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml logs --tail=200 vllm
nvidia-smi
```

## 4.2 方案 B：systemd vLLM + Docker API（无网络可落地）

### 4.2.1 启动模型服务

```bash
sudo systemctl enable --now qwen3-30b-a3b.service
sudo systemctl is-active qwen3-30b-a3b.service
curl -sS http://127.0.0.1:8001/v1/models
```

### 4.2.2 启动数据库与 API

```bash
docker compose --env-file /opt/clinic-sim/.env -f src/infra/compose/prod-b.yml up -d --build --pull never postgres minio api
```

### 4.2.3 环境变量要求

- `LLM_BASE_URL=http://<B_SERVER_IP>:8001`（或 API 可达地址）
- `LLM_MODEL=qwen3-30b-a3b`

## 4.3 方案 C：离线镜像导入

联网机器执行：

```bash
docker pull vllm/vllm-openai:latest
docker pull postgres:15-alpine
docker pull minio/minio:latest
docker save -o clinic-sim-images.tar vllm/vllm-openai:latest postgres:15-alpine minio/minio:latest
```

B 服务器执行：

```bash
docker load -i clinic-sim-images.tar
```

然后按方案 A 启动。

## 6. 推荐发布顺序（生产）

1. 本地构建并打包前后端发布物
2. 优先部署并验收 B（健康检查、模型可用、迁移完成）
3. 再部署 A（上传前端 + 并入 Nginx + reload）
4. 端到端冒烟：登录 → 建会话 → 对话 → 提交诊断
5. 观察日志与关键指标后再切主流量

## 7. 回滚总方案

### 7.1 配置回滚

- 回退 `.env` 到上一版本
- 重点检查：`LLM_BASE_URL`、`LLM_MODEL`、`DATABASE_URL`、`JWT_SECRET`

### 7.2 模型方式回滚

- Docker → systemd：先停 vllm 容器，再 `systemctl enable --now qwen3-30b-a3b.service`
- systemd → Docker：先 `systemctl disable --now qwen3-30b-a3b.service`，再起 vllm 容器

### 7.3 A 服务器回滚

- 回传上一版静态文件
- 回滚 Nginx 增量配置并 reload

## 8. 常见故障与定位

### 8.1 `:8001/v1/models` 连接拒绝

- 模型服务未监听 8001
- 排查：`ss -lntp | grep :8001`、`docker compose logs vllm`、`journalctl -u qwen3-30b-a3b.service`

### 8.2 API 可用但对话失败

- 大概率 `LLM_MODEL` 与 `id` 不一致
- 先校准模型名，再重启 API

### 8.3 端口/GPU 冲突

- Docker 与 systemd 同时起了 vLLM
- 停掉其中一种后重启另一种
